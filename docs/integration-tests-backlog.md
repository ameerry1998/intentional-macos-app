# Integration / Automated Tests Backlog

**Date:** 2026-05-06
**Status:** Living backlog. Update entries as tests are written or supplanted.
**Scope:** Automated tests to add across all three repos (`intentional-backend`, `intentional-macos-app`, `puck-ios`).
**Sister doc:** [`docs/test-plan-2026-05-05.md`](test-plan-2026-05-05.md) — manual / smoke checklist for the 2026-05-05 redesign. This backlog is the *automated* counterpart: every entry is a test someone could sit down and write tomorrow.

## Why this exists

The May 2026 redesign added 13 slices of new infrastructure (entitlements + Stripe webhooks, Focus Mode rename, distraction budget cross-device, schedule projection, app taxonomy, Focus Lock master toggle, Strict Mode constraints, AI→budget rewire, block lifecycle strictness gate, Mac sidebar 8→5, iOS tabs 6→3, Puck alarm-only flag, cleanup checklist). Coverage today is uneven — the backend has good coverage of older endpoints but only smoke-tested the new slices; the Mac has unit tests for older subsystems (Bedtime, ConfidenceGate, IntentionStore) but nothing for the new entitlement / banner / dual-write / strictness-gate code; iOS has tests for IntentionStore + IntentionalAPIClient but nothing for the new EntitlementGate / Landing / alarm-only flag.

Beyond new-slice coverage, the [Known Bug Fixes section of `CLAUDE.md`](../CLAUDE.md) documents a stack of production bugs (AppleScript-on-main UI freeze, Chrome SIGKILL of native messaging hosts, MLX fail-open, queued-project-session not auto-activating, etc.). Every one of those is a regression class worth pinning to a test.

This backlog targets **15–25 tests per repo**, ordered P0 → P2.

## Conventions

- **Layer**: `unit` (no I/O), `integration` (process boundary, DB, file I/O), `e2e` (end-to-end across processes / devices)
- **Effort**: `S` < 30 min, `M` 30 min – 2 h, `L` 2 h+
- **Priority**: `P0` write before next slice ships, `P1` write in the next sprint, `P2` write whenever
- **Sample assertions** are pseudo-code — adapt to pytest / XCTest idioms when writing

## Notes on infrastructure

- **Backend**: pytest already exists with a `_FakeDB` fixture in `tests/test_intentions.py` that's reused across the suite. Adding the new tests in this backlog needs no new dependencies. The fan-out test (B-08) and the migration idempotency test (B-23) are the only ones that may justify a real Supabase test instance — manageable via the local Supabase CLI's test DB. Concurrency tests (B-12 / B-13) need an async transaction-aware fixture; consider adding a thin `tests/_fixtures/async_db.py` helper before writing them.
- **Mac**: XCTest target `IntentionalTests` exists with 12 test files (Bedtime, ConfidenceGate, ConstraintEvaluator, FocusSession, IntentionStore, etc.). The pattern is in-process unit tests using fakes. Adding the entitlement / banner / strictness-gate tests requires no new infra. **Follow-up needed**: there is no mock URLProtocol harness for testing `BackendClient` HTTP behavior. Add a `MockURLProtocol` helper before writing M-04, M-06, M-07. Estimate 1 h to introduce.
- **iOS**: XCTest target `PuckTests` exists with 10+ test files including `IntentionalIntentionsClientTests` which already injects a mock URLSession via `IntentionalAPIClient.makeForTests(...)`. Reuse that pattern for the entitlement client tests (I-02, I-03, I-13, I-20, I-24). View-layer tests (I-09, I-10, I-11, I-19, I-25) need either ViewInspector (third-party SPM lib) or snapshot testing — flag adopting one of those as a one-time investment before writing the view-layer entries.
- **Dashboard (`dashboard.html`)**: there is **no JavaScript test infrastructure** yet for the embedded dashboard. The dashboard now has user-facing entitlement banner code, Focus Mode editor, and budget meter — none unit-tested. **Follow-up**: pick a JS test runner (Vitest is the obvious choice given how the codebase uses lightweight tooling), establish a `dashboard/__tests__/` dir with a single trivial test wired to CI, and then plan a separate dashboard test backlog. Without this, every dashboard change is hand-tested.
- **CI**: confirm the backend test suite runs in GitHub Actions on every PR. Mac + iOS XCTest do not run in CI today (per repo state). Setting up `xcodebuild test` on macOS GitHub runners for both is a P1 follow-up — the tests are valuable only if they actually run.
- **Coverage targets**: aim for ≥ 80% on the new slice 1–13 code paths within 2 sprints. The entitlement, banner, fan-out, strictness-gate, and PUCK_ALARM_ONLY paths are the load-bearing ones — make those P0 entries above the gate to merging.
