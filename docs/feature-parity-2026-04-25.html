TL;DR

You don't have feature parity, and you don't need it. The two apps have different jobs: the Mac is the enforcer (AI scoring, screen capture, browser blocking), and the iPhone is the trigger + bedtime + alarms. The problem isn't that features are missing on each side — it's that shared concepts (partner, lock state, settings, pucks, sessions) live in five different places and don't sync.

  • What actually overlaps — partner email, lock mode, focus modes, sessions, settings. Of those, only sessions actually sync (via the backend WebSocket). The other four diverge per device.
  • What only the Mac does — AI relevance scoring, content safety, earned browse, scheduled work blocks, browser extension, full strict-mode lockdown, partner backend integration.
  • What only the iPhone does — NFC puck reads, alarms, bedtime mode + Shortcuts, weekly habit goals, wake streak.
  • The single biggest gap — the iOS Partner tab is UI-only. Tap "Send invite," nothing reaches your accountability partner. Mac has the full backend integration. This needs to ship before either app feels real.

Where shared concepts diverge today

Concept

Partner email

Mac stores it in onboarding_settings.json + backend. iOS stores it in @AppStorage("partnerEmail") only. If you change it on one app, the other doesn't know.

Fix: iOS reads/writes through GET/PUT /partner like the Mac does. One source of truth = backend.

Concept

Lock mode

Mac respects lock_mode from backend (none / partner / self). iOS doesn't read it at all. You can be partner-locked on Mac and unrestricted on iOS.

Fix: iOS fetches lock_mode on launch; if "partner" or "self", surfaces lock UI on Settings tab + gates emergency-unbrick toggle.

Concept

Focus mode / block

iOS has on-demand modes (FocusMode SwiftData). Mac has scheduled blocks (FocusBlock + Deep Work / Focus Hours / Free Time). Same word, different model.

Fix: pick one canonical shape. The Mac's block model is richer; iOS modes can map onto it as "named templates that produce a block when activated".

Concept

Pucks

iOS has RegisteredPuck SwiftData. Mac doesn't have a puck registry — it just receives focus signals. If you re-pair a puck on iOS, Mac never finds out.

Fix: iOS posts puck registry to backend. Mac displays "Pucks registered to your account" in dashboard read-only.

Concept

Sessions

Mac has FocusSession with rich data (focus score, intent, recovery count, app breakdown). iOS has its own FocusSession SwiftData (simpler). They both end up in POST /sessions but the iOS data is thinner.

Fix: align the wire format. iOS doesn't need to fill every field, but the schema should match.

Concept

Onboarding

iOS: 4-question quiz → ResultView → puck pairing. Mac: 6-step wizard → theme → extension → blocked platforms → partner → lock. Two completely different first-runs.

Fix: pick a shared first-run flow. Quiz can be iOS-only (it's about reclaiming time), but partner setup + lock-mode choice should be identical screens.

What this DOESN'T mean

You don't need to make iOS and macOS feature-equivalent. The two apps have different jobs:

  • iPhone: the trigger. Tap a puck → start a session. Wake up with a smart alarm. Sleep with bedtime mode. Show the partner state. Show the Mac's status. Be the always-with-you remote control.
  • Mac: the enforcer. The work happens here — AI scoring, browser blocking, screen monitoring, the pill timer, the rituals. The Mac is where you get work done, the iPhone is the controller.

The unification work is about shared state (partner, lock, settings, sessions) flowing through the backend, not about copying features sideways.
