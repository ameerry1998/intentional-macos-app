📋 The convention

Two layers, two purposes. Use the right one for the right job, and keep them separated by file extension and naming.

LayerFormatFilename patternWhat it's for
Reference Markdown MD docs/SUBSYSTEM_NAME.md (UPPER_SNAKE_CASE) Evergreen source of truth. Updated when behavior changes. Rendered on GitHub. Source of truth for "how does this work right now."
Snapshot HTML HTML docs/topic-YYYY-MM-DD.html (lowercase-kebab + date) Point-in-time visual report — audit, run log, decision doc, sprint plan. Never edit an old one — write a new one with a new date if state has changed.
Mockup HTML HTML docs/topic-vN-variant.html (no date, has version) Visual design exploration. May or may not become the chosen direction. Keep around as reference.

When you write something new, ask:

  • "Will this still be true in 6 months?" Yes → Markdown. No → dated HTML.
  • "Should this get diffed in PRs?" Yes → Markdown. No → HTML.
  • "Will I want to skim this in Chrome?" Yes → HTML. No → Markdown.

When you add a new HTML report:

  1. Save it as docs/<topic>-YYYY-MM-DD.html
  2. Add a card to this index.html in the appropriate section (Reports & audits, or Mockups)
  3. Use the existing CSS variables and pill component classes — keeps everything visually consistent
  4. Never edit an old dated report. If the state has changed, write a new one and link from the old one.
The HTML reports are most useful when they're dense: status pills, color-coded comparisons, ranked lists, tables. Save the prose for Markdown.
